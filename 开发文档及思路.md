## 开源Excel处理工具设计文档

### 一. 定义IO边界

#### 1. 明确输入边界

1. 基础输入: 本地文件路径
2. 高级输入: 二进制流输入

#### 2. 明确输出边界

1. 定义一个数据类, 比如ExcelChunk; 每次处理, 输出的是一个List[ExcelChunk], 每个Chunk包含:
   - chunk_id: str 分块编号
   - metadata: Dict[str, Any]这个chunk的上下文信息(比如所属工作表的表名, 额外提取到的信息等)
   - formatted_context: str 可以直接作为prompt传入LLM的字符串(已经实现, 见excel_content_formatter)
   - raw_data: List[Dict] 原始的Python字典数组

### 二. 是否集成LLM服务

当用户使用这个库的时候, 有两种情况:

#### 1. 使用这个库对Excel表格进行纯粹的数据处理

- 针对这种情况, 只需要 输入Excel -> 输出List[ExcelChunk]

#### 2. 用户懒得写并发调用LLM的代码, 重试逻辑等

- 用户传入一个实现我定义接口的LLM Service 类

针对这种情况, 要有一个明确的责任划分, 责任表如下

| 功能           | 我的库负责 | 调用方负责 |
| :------------- | :--------- | :--------- |
| **并发控制**   | ✅ 负责     | ❌ 不负责   |
| **错误重试**   | ✅ 负责     | ❌ 不负责   |
| **负载均衡**   | ❌ 不负责   | ✅ 负责     |
| **超时阻断**   | ✅ 负责     | ❌ 不负责   |
| **请求大模型** | ❌ 不负责   | ✅ 负责     |

### 三. 对Excel处理的方法

#### 1. 消除合并单元格

将选择权交给用户

- action="fill_forward": 自动向下/向右复制主格值
- action="top_left": 保持原样(主单元格有值, 其它为None)
- action="tag": 注入自定义占位符

#### 2. 表格块检测

由于一张sheet中可能会存在多张表（如：上面是客户信息表，中间是零件明细表，下面是汇总签名）, 所以要对表格块进行检测; 现在有两种候选方法:

- 方法A: 连通域检测: 将Excel映射为一张二维图像, 使用图像处理中的连通域标记算法
- 方法B: 行列投影法: 计算二维布尔矩阵在X轴和Y轴的投影直方图, 找到连续的0作为切分刀, 递归或者循环切割

根据合并单元格策略, 我认为应该以方法A为核心, 方法B为辅助, 同时使用双联路架构

首先为什么不用B方法: 如果使用B方法, 并且action="top_left", 那么当遇到一个5行的合并单元格时, 第2~5行全都变成了None, 如果使用B方法, 算法会认为这里出现了断层, 从而把表格切断.

架构: 

- 链路1: 结构探测层, 找表格的包围盒; 在这一层, 要完全无视用户选择的action(合并单元格处理)参数, 在内存中构建一个用于检测边界的矩阵
  - 基础映射：有任何实实在在的数据（包括数字、文本），矩阵填 1，空单元格填 0。
  - 合并单元格修补: 无论用户选什么, 只要是合并单元格的覆盖区域, 哪怕除了左上角全为空, 在矩阵中也要将整个区域强行填充为1
  - 连通域计算: 在矩阵上跑连通域检测算法
- 链路2: 数据渲染层: 在链路1中已经拿到所有表格块的精确坐标, 在这一层开始提取真实的数据, 在这一层要严格执行用户传入的action参数
  - 提取Box对应的真实数据区域
  - 如果用户选择top_left, 则保留None
  - 如果选择fill_forward, 则补齐值

#### 3. 通用表头识别算法

- 启发式打分机制: 遍历每个block的前5行, 给每一行打分:
  - 特征1(密度法): 该行非空单元格占比, 满格行得分高(+30), 稀疏行得分低
  - 特征2(类型突变法): 如果该行全是字符串, 而下一行出现了int, float, date等, 则说明该行大概率是表头(+40)
  - 特征3(Style样式): 读取单元格样式。表头通常有明显的特征：加粗 (Bold)、居中 、背景色填充 、下边框等。如果某一行具备这些特征，且下一行没有，这行+20。
  - 综合判定: 得分最高且超过阈值的行, 及其上方所有行[可能有多行合并表头(存疑)]定义为header区域
- 兜底机制: 增加可选模式(use_llm_layout_analyzer=True), 将当前Block前10行处理成MD字符串, 发送给一个快速的小模型; 如果开启此模式则接上面的LLM服务
- 提供用户自定义表头特征, 如包含"xx","yy","zz"等, 做匹配

#### 4. 表格切片

可以提供几种策略供用户进行选择, 比如:

- 策略A: 固定行数切片: 如固定x行(行数不包含表头)为一批切片, 若最后一批行数<y, 则合并到上一批
- 策略B: 近似token取值, 如1字符=0.5token
- 注: 无论哪种策略, 每个chunk都要带上表头, 行数切片策略中表头不包含在行数内, token切片策略中行数包含在token内

### 四. Excel处理流程

基于你提供的设计文档前三部分，以及现有的 `excel_context_formatter.py` 和 `llm_service.py` 源码，我为你编写了第四部分：**Excel处理流程**的设计文档。

这部分文档将抛弃原代码中为了兼容旧版 CLI 工具而做的过度定制（如：子进程脚本注入、全局 `stdout` 劫持、硬编码的日志等），将核心的表格解析能力重构为一个标准化的、适合作为开源第三方库的流水线（Pipeline）。

为了实现输入 (Path/Bytes) 到输出 (`List[ExcelChunk]`) 的转化，并将第三部分的复杂策略（合并单元格、分块、表头识别）解耦，我们将整个核心处理流程设计为 **5个标准的 Pipeline 阶段**。

#### Phase 1: 安全预检与文件加载 (Security & Loading)

为了保证开源库在生产环境的鲁棒性，防止 OOM (内存溢出) 和“Excel炸弹”，在正式解析前必须进行物理层面的检查。

- **1.1 格式路由**: 识别输入是 `.xlsx` (使用 `openpyxl` / `calamine`) 还是 `.xls` (使用 `xlrd` 或转换逻辑)，若是二进制流则写入内存临时文件 (`io.BytesIO`)。
- **1.2 维度预检 (Dimension Check)**: 继承原代码中 `check_sheet_dimensions_before_load` 的优秀设计。不加载整个文件，而是直接读取 ZIP 结构中的 XML 的 `<dimension>` 标签，如果行列数超过阈值（如超过10万行），直接阻断或触发降级策略。
- **1.3 状态过滤**: 根据用户的全局配置，识别并决定是否跳过隐藏的工作表 (Hidden Sheets) 和隐藏行/列 (Hidden Rows/Cols)。

#### Phase 2: 结构探测与区块划分 (Structure Detection)

对应第三部分“表格块检测”的**链路1**，此阶段不在乎单元格的具体内容，只在乎数据的“形状”。

- **2.1 布尔矩阵映射**: 扫描当前 Sheet 实际使用的范围，构建二维布尔矩阵。有数据（或属于受支持的合并单元格范围）为 `1`，空单元格为 `0`。（继承并升华原代码 `find_data_boundary` 的边界收束逻辑）。
- 2.2 表格块切割 (Block Segmentation):
  - 运行连通域检测算法（主要）与投影直方图算法（辅助）。
  - 将单张 Sheet 中上下或左右分布的多个独立子表（如：表A是基础信息，表B是明细），切割成独立的数据包围盒 (Bounding Boxes)。
  - *输出: `List[BoundingBox]` (例如 Box1: A1:G10, Box2: A15:H50)*。

#### Phase 3: 数据渲染与合并单元格消解 (Data Rendering & Resolution)

对应“表格块检测”的**链路2**。对 Phase 2 产出的每一个 `BoundingBox` 进行真实数据提取。

- **3.1 基础类型清洗**: 读取单元格的值，进行基础的格式化。（继承原代码的 `format_cell_value`，如将 `1.0` 抹平为 `1`，处理基础的 Datetime 格式）。
- 3.2 合并单元格消解 (Merged Cell Resolution):
  - 读取 Box 内的合并单元格拓扑信息 (`merged_cells.ranges`)。
  - 严格按照用户传入的 `action` 策略 (`fill_forward`, `top_left`, `tag`) 渲染子单元格的值。
- **3.3 生成二维字典数组**: 将提取到的数据组装为纯 Python 的 `List[List[Any]]` 或 `List[Dict]`，此时已经彻底脱离了底层 Excel 引擎的依赖。

#### Phase 4: 表头识别专家系统 (Header Recognition)

对 Phase 3 产出的纯净数据块进行表头分离。

- **4.1 启发式打分 (Heuristic Scoring)**: 提取当前数据块的前 NNN 行，综合运行**密度法**、**类型突变法**、**样式特征法**进行评估，切分出 `Header` 区域和 `Body` 区域。
- 4.2 LLM 分析器 (可选):
  - 若 `use_llm_layout_analyzer=True` 且传入了实例化的 `LLMService`（见第二部分）。
  - 将前 NNN 行转为基础 Markdown 表格，并调用 `LLMService.chat()`。
  - Prompt 示例: *"请分析以下电子表格的头部，返回表头占据了前几行，以JSON格式输出 `{"header_row_count": 2}` "*。
  - 根据大模型的返回结果精准切割表头。

#### Phase 5: 上下文切片与组装 (Chunking & Formatting)

将带着表头的数据体，切分为可以喂给最终大模型的 `ExcelChunk`。

- 5.1 切片执行 (Chunking):
  - 根据选定的策略（固定行数 或 Token 预估），将 `Body` 区域切割为多个片断。
  - 为每个片断赋予全局逻辑行号（`logical_idx`，保证大模型能够准确定位这是第几行数据）。
- 5.2 字符串格式化 (Context Formatting):
  - 继承原代码中 `format_excel_context` 的逻辑。
  - 组装格式：`表名` + `是否折叠/隐藏` + `表头栏 (列1 | 列2)` + `数据行 (行11: 值A | 值B)`。
  - 生成 `formatted_context`。
- 5.3 封装输出 (Construct Output):
  - 将上述所有信息封装为最终的数据类对象，返回给调用侧。

### 五. 项目目录结构设计

```text
llm_excel_parser/
├── __init__.py                # 暴露给用户的顶级API (如 process_excel, ExcelChunk)
├── core/                      # 【核心层】模型、协议与异常定义
│   ├── __init__.py
│   ├── datatypes.py           # 数据类定义 (ExcelChunk, BoundingBox)
│   ├── interfaces.py          # 接口定义 (LLMServiceProtocol)
│   ├── exceptions.py          # 自定义异常 (FileHugeError, OverDimensionError等)
│   └── enums.py               # 枚举类型 (MergeAction, ChunkStrategy)
├── pipeline/                  # 【流水分层】核心的 5 Phase 实现
│   ├── __init__.py
│   ├── orchestrator.py        # 流水线调度机 (串联Phase 1~5)
│   ├── phase1_loader.py       # 安全预检与文件加载 (XML维数解析, 驱动路由)
│   ├── phase2_detector.py     # 结构探测 (构建布尔矩阵, 连通域分离算法)
│   ├── phase3_renderer.py     # 数据渲染 (提取Box数据, 执行合并单元格策略)
│   ├── phase4_header.py       # 表头识别 (启发式特征打分, 衔接LLM降级分析)
│   └── phase5_chunker.py      # 切片与组装 (行数/Token切片, Markdown/CSV格式化)
├── strategies/                # 【策略层】将复杂算法解耦，方便未来扩展
│   ├── __init__.py
│   ├── chunking.py            # 切片策略 (FixedRowChunker, TokenChunker)
│   └── unmerge.py             # 合并格策略 (FillForward, TopLeft, Tag)
├── utils/                     # 【工具层】
│   ├── __init__.py
│   ├── matrix_algo.py         # 纯Python/Numpy实现的二维矩阵工具 (BFS连通域, 投影定位)
│   ├── excel_style.py         # 样式的辅助解析 (提取加粗、背景色等)
│   └── concurrency.py         # 并发、重试、超时控制器 (包装用户的LLMService)
├── config.py                  # 全局默认配置 (启发式分数阈值, 最大行列数保护限制等)
└── prompts.py                 # 内置的系统级Prompt (如用于Phase 4分析表头的Prompt)

# --- 根目录其他规范文件 ---
├── tests/                     # 单元测试 (需覆盖合并单元格、表头打分等核心场景)
├── examples/                  # 示例代码 (如何传入LLMService, 各种action的对比)
├── pyproject.toml             # 依赖与打包配置 (openpyxl, tenacity等)
└── README.md                  # 开源项目说明文档
```
